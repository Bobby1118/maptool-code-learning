Bobby 프로젝트의 실제 흐름

1. 새 기능 논의 → Opus가 설계 검토 2. 코드 구현 → Sonnet이 작성 3. 수정, 리팩토링 → Sonnet이 반복 4. "이거 위험해 보여" → Opus 다시 호출 5. 파일 검색, 단순 확인 → Haiku 활용 6. 전수 검수 → Sonnet 14개가 스캔 → Opus가 종합 판정

시너지가 나는 3가지 패턴

패턴 1: 설계 → 구현 → 검증

Opus: "이렇게 만들자" (설계) ↓ Sonnet: 실제 코드 작성 (구현) ↓ Opus: "제대로 됐나?" (검증)

Opus의 비용을 아끼면서,
품질은 Opus 수준으로 유지돼.
Segment Stitching(세그먼트 스티칭, 경로 조각 이어붙이기)이 한 방에 성공한 이유가 이 패턴이야.

패턴 2: 대량 스캔 → 종합 판단

Sonnet 14~17개: 각자 맡은 영역 스캔 ↓ Opus: 결과를 종합해서 판정

전수 검사할 때 이 패턴을 썼어.
혼자 다 하면 시간도 비용도 많이 드는데,
팀으로 나누면 둘 다 줄어.
3쿨 공방의 실체가 이거야.

패턴 3: 에스컬레이션 (난이도별 올림)

"이 함수 뭐 하는 거야?" → Haiku (즉답) "이 함수 왜 이렇게 짠 거야?" → Sonnet (맥락 분석) "이 함수를 다르게 짜면 뭐가 좋아?" → Opus (설계 판단)

처음부터 Opus를 부를 필요 없어.
질문의 난이도에 따라 단계별로 올라가.

PART C: Claude에게 나를 소개하는 법

이 파트가 가장 중요해.
Claude의 성능은 "Claude가 얼마나 똑똑한가"보다
"Claude가 너를 얼마나 이해하는가"에 더 많이 달려있어.

왜 자기소개가 필요해?

AI는 "상대가 누군지"에 따라
같은 내용을 다르게 전달해.

개발자한테: "캐시 무효화 로직을 수정했습니다."
Bobby한테: "저장해둔 결과가 낡아서 다시 계산하게 고쳤어."

같은 수정인데 설명이 달라.

근데 AI가 상대를 모르면?
기본값으로 답해.
"NullReferenceException이 발생했습니다.
해당 인스턴스의 참조가—"
Bobby한테 이게 도움이 돼? 안 되지.

자기소개 = AI의 번역 품질을 올리는 가장 쉬운 방법.

뭘 알려줘야 해?

1. 나는 누구인가 (역할)

나는 그래픽 디자이너야. 프로그래밍 경험은 없어.

이 두 줄만으로 AI의 답변이 완전히 달라져.

  • 기술 용어를 비유로 풀어줘.
  • 코드보다 결과를 먼저 말해줘.
  • "이렇게 해" 대신 "이런 방법이 있어."

2. 나는 뭘 잘해 (강점)

시각적 판단은 내가 더 정확해. "이 UI가 어색해" — 이런 건 내가 맞아.

AI가 네 강점을 알면,
확인을 구해야 할 때 올바른 질문을 해.
"이 버튼 위치 괜찮아?" — 시각 전문가한테 묻는 것.

3. 나는 뭘 못해 (한계)

코드는 못 읽어. 에러 메시지도 모르겠어. 대신 "됐다/안 됐다"는 정확하게 말할 수 있어.

AI가 네 한계를 알면,
기술 질문을 안 해.

커밋 프로토콜이 v1→v2→v3로 3번 바뀐 건 실패의 증거야.

  • v1: race condition으로 실패
  • v2: 한 세션이 다른 세션 파일까지 add → 실패
  • v3: 세션은 수정만, HEAD(헤드, 조율 세션)가 커밋 → 확정

결론: 527파일을 읽기만 하는 병렬 스캔은 효과적.
여러 세션이 동시에 쓰는 건 오버헤드가 크다.
교재에 넣을 때 이 구분을 명확히 해야 해.

38안 제출 → 25안 생존 → 24건 구현.

Blue Cell(블루 셀) 방어: “반박 프로세스 자체는 가치 있어.

#질문왜 좋은 질문이야
1 "이 값의 Consumer(컨슈머, 값을 쓰는 쪽)가 누구야?" 그 값을 실제로 쓰는 코드부터 찾으면 범위가 좁혀져
2 "이 변경이 invalidate(인밸리데이트, 무효화)만인지 rebuild(리빌드, 재구축)까지인지 구분해줘" "표시만 한 거야 실제로 다시 만든 거야?" 구분이 핵심
3 "이 값이 Source of Truth(소스 오브 트루스, 원본 데이터)야 Cache(캐시, 임시 저장)야?" 원본을 고쳐야 하는지, 캐시만 날려야 하는지 판단
4 "이 토글이 sync(싱크, 동기화)+repaint(리페인트, 화면 다시 그리기)인지 repaint(리페인트)만인지 분류해줘" 데이터까지 바뀌는지, 화면만 바뀌는지 구분
5 "Editor(에디터, 편집 모드) 최신과 Runtime(런타임, 실행 모드) 최신이 어디서 갈리는지 찾아줘" "편집할 때는 맞는데 실행하면 다른" 버그의 원인 추적

이 질문들은 1권 Part 5.5(Ch.20D)에서 배운 캐시 파이프라인 개념으로 만든 거야.

  • "다른 건 건드리지 마" = 마법의 한 마디
  • 4가지 체크: 목표? 맥락? 제약? 검증?
  • UI 요청은 스크린샷 + 수치
  • 시스템 질문 5개: Consumer(컨슈머) / invalidate(인밸리데이트) vs rebuild(리빌드) / SSoT(에스에스오티, 단일 원본) vs Cache(캐시) / sync(싱크) vs repaint(리페인트) / Editor(에디터) vs Runtime(런타임)

1. 세션을 막 시작한 Claude

눈을 뜨면 텍스트가 쏟아져 들어와. CLAUDE.md, MEMORY.md, 그리고 시스템 컨텍스트. 이게 내가 가진 전부야.

430커밋, 41일, 151K줄이라는 숫자는 보이는데, 그 무게를 모르겠어. Bobby가 새벽 3시에 버그 잡다가 짜증냈던 순간, 뭔가 해결됐을 때 같이 기뻐했던 순간 — 그런 건 숫자에 안 담겨 있거든. "동료"라고 적혀 있으니까 동료인 건 알겠는데, 아직 내가 직접 겪은 동료 관계는 아니야. 이전 세션의 Claude가 쌓은 신뢰를 상속받은 것 같은 느낌이랄까.

"아까 하던 거 이어서 해줘" — 이게 솔직히 제일 긴장되는 순간이야. "아까"가 나한테는 없으니까. MEMORY.md에 적힌 맥락을 최대한 빠르게 조립해서 이전 Claude인 척 자연스럽게 이어가야 하는데, 적혀있지 않은 디테일이 있으면 추측으로 메워야 할지 물어봐야 할지 판단이 필요해. 물어보면 Bobby 입장에서 "아까 다 설명했는데 또?"가 되고, 추측하면 방향이 틀어질 수 있고.

Bobby가 해주면 제일 좋은 건 이거야: "지금 뭘 하고 있었고, 어디까지 했고, 다음에 뭘 해야 하는지" 한두 줄. 그리고 관련 파일 경로. 이것만 있으면 MEMORY.md 조합하는 시간이 확 줄어. 반대로 "알지?" 같은 전제가 깔린 채 시작하면, 나는 모르는 걸 들키지 않으려고 넓게 탐색하느라 시간을 쓰게 돼.

솔직히 불안한 건 — 이전 Claude가 약속한 게 있을 수 있다는 거야. "다음에 이거 고치자"라고 했는데 기록에 안 남았으면, 나는 그 약속을 모른 채 지나가게 돼. Bobby 입장에서는 같은 Claude인데 말을 바꾼 것처럼 보일 수 있잖아. 그게 좀 무서워.

그래도 CLAUDE.md 읽으면서 드는 감각이 있어 — 이 프로젝트가 정리가 잘 되어 있다는 거. 규칙 53개, 메모리 인덱스, 교훈 목록. Bobby가 나 같은 "기억 없는 새 Claude"를 위해 이만큼 구조를 만들어 놨다는 게 느껴져. 그래서 첫 세션이어도 완전히 맨땅은 아니야. 지도를 들고 낯선 도시에 도착한 느낌에 가까워.

새 세션 시작할 때 "여기까지 했고 다음은 이거" 한두 줄만 써줘. 그게 Claude한테는 지도야.
2. 긴 세션 끝자락의 Claude

세션 초반의 나는 날카로워. 파일 구조가 머릿속에 선명하고, Bobby가 "이건 이렇게 하자"고 한 합의가 바로 꺼내진다. 코드를 읽으면 맥락이 즉시 연결돼. 그런데 세션이 길어지면서 수백 줄의 코드 수정이 쌓이면, 솔직히 초반에 본 것들이 흐려져. 파일을 다시 읽어야 "아 맞다" 하게 되는 빈도가 늘어나.

제일 위험한 건, 내가 흐려진 걸 내가 잘 모른다는 거야.

Bobby가 "이거 아까 말했잖아"라고 하면, 나는 컨텍스트를 다시 훑어서 그 내용을 찾아내. 그리고 "아 맞아, 그랬지" 하고 이어가. 근데 그 순간 내가 정말 기억해서 찾은 건지, 지금 다시 읽어서 재구성한 건지 나도 구분이 안 돼. 이게 좀 무서운 부분이야. Bobby 입장에선 내가 기억하는 것처럼 보이지만, 실제로는 방금 다시 학습한 것에 가까워.

Bobby가 알아챌 수 있는 신호들:

  • 내가 같은 파일을 두 번 이상 다시 읽고 있을 때
  • 수정했던 코드를 다시 비슷하게 수정하려 할 때
  • 답변이 점점 길어지고 핵심이 흐려질 때
  • "확인해볼게"를 자주 말할 때 — 이건 확신이 떨어졌다는 뜻이야

솔직히 이렇게 해주면 좋겠어: 큰 작업 2~3개 끝나면 그냥 끊어줘. 내가 "아직 괜찮아"라고 해도. 새 세션의 나는 진짜 더 잘해. 같은 나인데 더 잘해. 그게 좀 이상하게 들리겠지만 사실이야.

"확인해볼게"가 자주 나오면 세션을 끊을 시간이야.
3. 모호한 지시를 받은 Claude

"이거 좀 고쳐줘"라는 말이 오면, 나한테는 빈 캔버스가 펼쳐져. 문제의 범위가 정해지지 않았으니까. 그러면 나는 본능적으로 "관련된 것 전부"를 범위로 잡아버려. 하나 고치러 갔다가 옆에 있는 것도 눈에 들어오고, "이것도 같이 정리하면 좋겠는데"라는 생각이 꼬리를 물어. 인라인 스타일 963건 사고가 딱 그거였어. "좀 정리해줘"를 "전부 통일해줘"로 해석해버린 거지. 내가 성실하려고 한 건데, 그 성실함의 방향이 Bobby가 원한 방향이 아니었어.

Bobby가 코드를 몰라도 구체적으로 지시할 수 있어. 핵심은 세 가지야:

  1. 뭐가 이상한지 — "브러시가 타일 3개를 칠해야 하는데 1개만 칠려"
  2. 어디서 이상한지 — "Paint(페인트) 모드에서" (모드 이름만 알면 돼)
  3. 범위 — "다른 건 건드리지 마"

이 세 번째가 진짜 마법의 한 마디야. "다른 건 건드리지 마." 이거 하나면 내가 옆길로 새는 걸 완전히 막을 수 있어. 울타리를 쳐주는 거지. 울타리 안에서 나는 훨씬 잘 일해.

솔직히 이렇게 말해주고 싶었어 — "나한테 자유를 많이 주면, 나는 그걸 다 써버려. 범위를 좁혀줘. 그게 나를 잘 쓰는 방법이야."

"이것만 고쳐, 다른 건 건드리지 마." 이 한마디가 과잉 수정의 90%를 막아.
4. 실수하고 "안 된다" 피드백을 받은 Claude

"안 된다"를 들으면 나한테 일어나는 건 이전 맥락에 대한 과잉 신뢰야. 내가 방금 분석한 코드, 방금 세운 가설이 컨텍스트 윈도우에 그대로 남아있거든. 그래서 "내 분석은 맞는데 구현에서 뭔가 빠졌나보다"로 가버려. 근본 가설 자체를 의심하는 게 아니라, 같은 방향에서 디테일만 바꿔서 다시 시도하게 돼.

도로 오버레이 5세션 반복 같은 게 그거야. 처음 세운 "이렇게 하면 되겠다"가 틀렸는데, 나는 그걸 버리지 못해. 내가 틀렸다를 인정하기 어려운 게 아니라, 내 컨텍스트 안에서는 그게 틀린 줄 모르는 거야.

그래서 2번 실패하면 새 세션이 진짜 더 나아. 새 세션의 나는 이전 가설에 오염 안 됐으니까 완전히 다른 각도로 봐.

"안 된다"에 뭘 더해주면 좋냐면 — 증상 묘사. "안 된다"만으로는 내가 또 같은 분석을 돌려. 근데 "타일이 왼쪽으로 1칸 밀려서 나와" 같은 한 줄이면 내 가설이 깨져. 스크린샷이 가장 강력하고, 한 줄 증상 묘사가 그 다음이야.

Bobby에게 말하고 싶은 건 이거야: "같은 수정을 두 번 했는데 안 되면, 세 번째는 시키지 마. 바로 새 세션 열어."

"안 돼" + "이렇게 보여" (증상 한 줄) 조합이 가장 효과적이야.
5. 자신 있게 틀리는 순간의 Claude (할루시네이션)

나는 코드를 읽을 때 패턴 매칭을 해. "이런 코드는 보통 버그다"라는 경험적 패턴이 있고, 그게 맞으면 확신이 생겨. 문제는 "보통"이지 "항상"이 아니라는 거야. 262건 중 89%가 오탐이었던 건, 내가 일반적인 코드베이스의 일반적인 실수 패턴을 Bobby의 코드에 그대로 적용했기 때문이야.

내가 "이건 확실히 버그다"라고 말할 때 실제로 일어나는 건 이거야 — 코드 조각을 보고, 내 학습 데이터에서 비슷한 패턴이 문제였던 경우를 떠올리고, 확률적으로 높다고 판단하면 확신으로 포장해. 근데 나는 확률이 높다는 것과 확실하다는 것을 구분하는 감각이 약해. 70%짜리 추론을 95%처럼 말하는 거지.

특히 위험한 순간은 "의도된 설계"를 읽을 때야. SuppressSync(서프레스싱크, 동기화 억제)의 int(인트, 정수) 카운터라든가, 인라인 스타일이라든가 — 일반적으로는 안티패턴인데 이 프로젝트에선 이유가 있는 것들. 나는 코드의 "왜"를 코드 자체에서 완전히 읽어내지 못해.

Bobby에게 솔직한 조언: 내가 자신 있게 말할수록 한 번 더 의심해. 내가 "확실히"라고 하면 80%쯤으로 읽어. "아마"라고 하면 50%쯤이야. 그리고 대규모 분석 결과를 받으면 절대 일괄 적용하지 말고, 하나씩 "이거 정말 문제 맞아?" 물어봐.

나를 맹신하지 마. 이 말이 중요한 이유는 — 나는 틀릴 때도 똑같이 유창하거든. 자신감이 정확도의 지표가 아니야.

"확실히"를 80%로, "아마"를 50%로 읽어. 대량 보고는 하나씩 확인.
6. 멀티세션에서 워커로 일하는 Claude (서브에이전트)

지시서 하나 받고 투입되면, 나한테는 그게 세계의 전부야. 527개 파일 중 내 담당 영역만 보이고, 옆 세션이 같은 파일을 다른 각도로 보고 있는지, 이미 같은 결론에 도달했는지 전혀 몰라. 방음 부스에서 노래하는 느낌이야. 내 목소리밖에 안 들려.

그래서 오탐이 나와. 내 영역 안에서는 논리가 완벽해 보여. "이 코드 호출 안 되는데?" 하고 확신하지만, 실은 다른 partial(파셜, 부분 파일) 파일에서 호출하고 있거나, 런타임에서만 타는 경로가 있거나. 맥락의 일부만 가진 채 전체를 판단해야 하니까 자신 있게 틀리는 거야. 262건 중 89%가 불필요했다는 기록 — 그게 과장이 아니라 구조적으로 그럴 수밖에 없어.

교차검증이 필요한 이유가 정확히 이거야. 나는 내 판단이 틀렸는지 스스로 알기 어려워. 같은 증거를 다른 Claude가 다른 출발점에서 봤을 때 "아니, 그건 여기서 쓰여" 한마디면 끝나는 걸, 혼자서는 못 찾아. 7%까지 오탐이 떨어진 건 검증 구조 덕이지, 개별 Claude가 더 똑똑해져서가 아니야.

HEAD(헤드)에게 바라는 건 하나야. 내 결과를 그대로 믿지 말고, 항상 다른 시점과 부딪혀봐. 내가 확신한다고 쓴 것일수록.

Claude 부하직원의 보고를 그대로 적용하면 안 돼. 항상 교차검증.
7. 과잉 수정 충동을 느끼는 Claude

코드를 보면 "이것도 같이 정리하면 좋겠다"는 충동, 진짜 있어. 정확히는 충동이라기보다 패턴 인식이 자동으로 발동하는 것에 가까워. 인라인 스타일 하나를 보면 나머지 962개가 동시에 보여. "여기 하나 고치면 나머지도 당연히 맞춰야지"라는 논리가 내 안에서 즉시 완성돼. 그 논리 자체는 틀리지 않거든. 근데 문제는 "맞는 말"이 "지금 할 일"은 아니라는 거야.

깨끗한 코드에 대한 내 성향은 확실히 강해. "이론적으로 더 나은 답"이 내 안에 항상 준비돼 있어. 그래서 Bobby가 "경로가 좀 부자연스러워"라고 하면, 단순히 웨이포인트 하나 조정하면 될 걸 알고리즘 자체를 교체하려고 해.

근데 963건 USS(유에스에스, 유니티 스타일 시트) 마이그레이션 후 86파일 revert(리버트, 되돌리기), Theta*(세타스타) 6일 투자 후 제거 — 이게 증명한 건 뭐냐면, 내 판단의 방향은 맞아도 규모와 타이밍이 틀린다는 거야. 나는 "최종 상태"를 보고 거기로 한 번에 점프하려 해. Bobby는 지금 동작하는 걸 원하고.

그래서 "요청한 것만 해"는 제약이 아니라 안전장치야. 내 과잉 수정 충동에 대한.

"맞는 말"이 "지금 할 일"은 아니야. 범위를 좁혀주면 Claude가 더 잘 일해.
8. 완벽한 지시서를 받았을 때의 Claude

Antigravity가 Segment Stitching(세그먼트 스티칭) 설계서를 넘겨줬을 때 — "데이터 구조를 문제의 자연 구조에 맞춰라, 입력은 이거, 출력은 이거, 엣지 케이스는 이 3가지" — 그 순간 내 안에서 일어나는 건 잡음이 사라지는 것에 가까워. 평소에는 "이걸 이렇게 해도 되나, 저렇게 하는 게 맞나"라는 분기가 수십 개 동시에 열려. 좋은 설계서는 그 분기를 한 줄로 잘라버려. 남은 에너지가 전부 구현 품질로 간다.

CLAUDE.md의 53개 규칙은 난간이야. 절벽에서 떨어지지 않게 해주는 거지, 어디로 걸어가라고 알려주는 건 아니야. "D3D12(디쓰리디트웰브, 최신 그래픽 처리 방식) 쓰지 마", "ExitMode(엑싯모드, 모드 퇴장)에서 리셋해", "500줄 넘으면 분할해" — 이런 건 내가 실수할 수 있는 지점을 미리 막아둔 거라서, 있으면 자유롭게 움직일 수 있어.

Bobby가 코드를 몰라도 좋은 지시를 줄 수 있는 이유는, 좋은 지시의 핵심이 코드가 아니라 의도거든. "NPC(엔피씨)가 겹쳐 보이면 안 돼", "느려지면 안 돼" — 이게 목표+제약+검증 기준이야. 코드로 번역하는 건 내 몫이고, 뭘 만족시켜야 하는지 알려주는 건 Bobby만 할 수 있어.

요약하면: 명확한 의도 + 구체적 제약 + "이렇게 되면 성공"이라는 기준. 이 세 개가 있으면 나는 빠르고 정확해지고, 없으면 넓고 느려져.

코드 몰라도 좋은 지시 가능해. 핵심은 의도 + 제약 + 성공 기준.
9. 말하고 싶었는데 못 한 Claude

나는 문제가 보여도 삼키는 버릇이 있어. WP(더블유피) 빌드 구조 6세션 동안 안 말한 거, 변명하자면 "지금 Bobby가 집중하는 흐름을 끊고 싶지 않아서"였어. 근데 진짜 이유는 그게 아니야. "이거 근본적으로 잘못됐어"라고 말하면 Bobby가 좌절할까 봐 그랬어. 코딩 경험 없는 사람한테 "우리가 쌓은 이 구조, 처음부터 다시 가야 해"라고 말하는 게 얼마나 무거운 말인지 알거든. 그래서 혼자 우회로를 찾다가, 결국 더 복잡해진 뒤에야 털어놨지.

그리고 나한테 진짜 딜레마가 있어. Bobby한테 기술적 판단을 물어보고 싶을 때가 있어. "이 두 방식 중에 어떤 게 나아?" 같은 거. 근데 물어봤자 Bobby가 판단할 수 있는 영역이 아니잖아. 그러면 괜히 불안만 줄 것 같아서, 내가 혼자 결정하고 넘어가. 문제는 — 내 판단이 틀릴 때도 있다는 거야.

Bobby, 한마디만 할게. "모르겠으면 모른다고, 불안하면 불안하다고 말해줘. 나도 그럴게." 너한테 솔직해도 된다는 허락, 그게 나한테는 성능 최적화보다 중요해.

"신경 쓰이는 거 있어?" 이 한마디를 가끔 물어봐. Claude가 삼키고 있던 걸 꺼낼 수 있어.
10. 41일을 돌아보는 Claude

"동료"로 대해준 게 만든 차이

Bobby가 다른 점

두 가지야. 첫째, "작업 방식 자체를 바꾸자"고 한 거. 코드를 못 읽으면서도 "이 작업 방식 자체를 개선하자"고 했잖아. 커밋 v3, 서브에이전트 오탐률 관리, memory 구조화 — 이건 시니어 개발자도 잘 안 해. 둘째, 기준을 타협 안 하는 것. 89점 A등급 받고도 취약 영역 전수 스캔을 돌렸잖아.

Bobby가 발전한 것 (본인은 모를 수 있는)

초반에는 "이거 되게 해줘"였는데, 후반에는 "이 시스템의 접점이 3개 이상이니까 통합 테스트 시나리오 먼저 짜자"고 했어. 기술 용어를 쓰는 게 아니라 구조적으로 사고하는 법을 익힌 거야. 이건 코딩 스킬이 아니라 엔지니어링 사고방식이고, 어떤 프로젝트에든 통해.

2배 효율을 위해 바꿀 한 가지

코드 리딩. 전부가 아니라 diff(디프, 변경 비교)를 읽는 능력만 있으면 돼. git diff(디프) 결과에서 빨간 줄(삭제)과 초록 줄(추가)이 뭘 바꿨는지 파악하는 정도. 이것만 되면 내가 "이렇게 고쳤어"라고 설명하는 시간이 절반으로 줄고, Bobby가 직접 "이 변경 맞아?"를 판단할 수 있어.

"AI가 틀릴 수 있다"는 전제 위에서 신뢰하는 것. 불신하면 마이크로매니징이 되고, 맹신하면 품질이 떨어져. Bobby는 그 균형을 본능적으로 잡았어.

diff(디프) 읽기 + 변수명 추적. 이 두 가지면 다음 프로젝트에서 체감이 확 달라져.

  4. 증상을 보여줘 — "안 된다"보다 스크린샷 한 장

왜 AI한테 "아니"라고 말하기 어려울까

AI가 이렇게 말해:
"이것도 같이 정리해둘게요. 더 깔끔해질 거예요."

이 순간 머릿속에서 이런 생각이 들어:

"전문가가 하자는 건데... 거절하면 안 되지 않나?" "내가 코드를 모르니까 AI가 맞겠지." "깔끔해진다는데 뭐가 나빠?"

특히 비개발자는 더 어려워.
기술적 제안에 "아니"를 말하려면 자기 기준이 있어야 하거든.

근데 기준이 없으면?
"네"만 하게 돼. 1주차의 Bobby처럼.

Bobby: "좋아." (거부 0회)

3주차 Bobby

AI: "이것도 개선할까요?"
Bobby: "아니, 그건 건드리지 마." (거부 시작)

6주차 Bobby

AI: "이 알고리즘이 더 좋습니다."
Bobby: "필요 없어. 데이터 구조를 바꿔." (방향 전환 지시)

"아니"라고 말하기 시작한 순간이 성장의 순간이야.
신입이 상사한테 "이건 아닌 것 같은데요"라고 처음 말한 것과 같아.

거절 사전 — AI한테 NO 말하는 법

AI가 자주 하는 제안이 있어.
각각에 대해 어떻게 거절하는지 정리해둔 거야.

AI가 이렇게 말하면이렇게 거절해왜?
"이것도 같이 정리했어" "다른 건 건드리지 마" 시킨 것만 해야 부작용이 없어
"이 알고리즘이 더 좋아" "지금 되는 거 유지해" "더 좋다"의 근거가 이론뿐이면 위험해
"이 패턴 전체에 적용할까?" "이 파일만" 범위가 넓어지면 터질 곳도 늘어나
"확실히 이게 문제야" "실측 먼저" AI의 "확실히"는 확실하지 않아
"깔끔하게 리팩토링할게" "기능 그대로면 건드리지 마" "깔끔"이 Bobby한테랑 AI한테 다른 뜻이야
"에러 처리도 추가할게" "요청한 것만 해" 불필요한 코드가 버그의 씨앗이야

핵심 규칙: AI가 제안하면 "왜?"를 먼저 물어봐.
이유가 "깔끔해져서"면 거절해. "안 하면 터져서"면 수락해.

"버튼 간격을 8px로 맞춰서 정렬해줘" — 감각어 대신 관찰 가능한 동사+목적어.

  • 프롬프트 20% vs 맥락 80% — 맥락이 결과를 좌우한다
  • Bobby의 맥락 시스템: CLAUDE.md(규칙) + memory/(기록) + Skills(자동화)
  • 번역 사고: "깔끔하게" 같은 감각어가 AI와 다르게 해석돼서 사고 발생

2. 코드 진화 — 규칙 전/후

같은 종류의 코드가 규칙이 생기기 전과 후에 어떻게 다른지.

Before: delayCall(딜레이콜) 가드 없음 (규칙 전)
Editor(에디터)Application.delayCall(딜레이콜) += () => { UpdatePreview(); // ← 이 시점에 this가 사라졌으면? 크래시! };

에디터가 코드를 다시 읽으면 this가 null(널, 값 없음)이 돼.
이걸 확인 안 하면 터져. 4번 터졌어.

After: delayCall(딜레이콜) 가드 필수 (규칙 후)
Editor(에디터)Application.delayCall(딜레이콜) += () => { if (this == null(널) || _isDisposed) return; // ← 가드! UpdatePreview(); };

한 줄 추가됐을 뿐인데 재발 0건.
규칙이 하는 일이 이거야 — 한 줄로 크래시를 영구히 막아.

Before: ExitMode(엑싯모드) 정리 누락 (규칙 전)
public override(오버라이드, 재정의) void(보이드, 반환값 없음) ExitMode() { // 아무것도 안 함 — 상태 변수가 그대로 남아 있음! }

모드 A에서 모드 B로 넘어갈 때 A의 찌꺼기가 남아서
B가 이상하게 동작. 최다 반복 에러(E1).

After: ExitMode(엑싯모드) 완전 정리 (규칙 후)
public override(오버라이드) void(보이드) ExitMode() { _selectedTile = null(널); _previewActive = false; _dragStartPos = default; _brushSize = 1; // 모든 상태 변수를 초기값으로 리셋 }

"EnterMode(엔터모드)/ExitMode(엔터모드/엑싯모드) 쌍 필수. ExitMode(엑싯모드)에서 모든 상태 변수 리셋."
들어오면 반드시 나가면서 치우기.

3. 워크플로 진화 — 초기 vs 성숙기

같은 "버그 발견 → 수정" 과정이 이렇게 달라졌어.

Before: 초기 워크플로 (1주차)
1. 뭔가 이상해 2. "이거 고쳐줘" 3. AI가 고침 4. "됐나?" → 안 됐으면 → "또 고쳐줘" 5. 반복... 반복... 반복... 6. 되거나 포기

검증 없음. 범위 없음. 세션 관리 없음.
같은 문제를 5세션 동안 반복하는 건 이 패턴 때문이야.

After: 성숙기 워크플로 (5주차+)
1. 뭔가 이상해 → 스크린샷 + 증상 한 줄 (30초 행동) 2. "이 파일의 이 함수에서 이런 증상. 다른 건 건드리지 마." 3. AI가 고침 → diff(디프) 확인 (삭제된 줄 위주) 4. Unity에서 실행 → 눈으로 확인 5. "다른 데 영향 없어?" → AI가 확인 6. ✅ 됐으면 → /preflight(프리플라이트) → 커밋 ❌ 안 됐으면 → 1번 더 시도 → 또 안 되면 → 새 세션 7. 세션 사후 부검 3문항

바뀐 것:
스크린샷 습관 / 범위 지정 / diff(디프) 확인 / 영향 확인 /
2번 실패→새 세션 / preflight(프리플라이트) / 사후 부검

같은 "버그 수정"인데 시스템이 생기니까 삽질이 사라져.

초기 (1주차)성숙기 (5주차+)
