PART 1

마인드셋

코드를 모르는 사람이 15만 줄을 만드는 이유

Ch.1 디렉터라는 자리

약 8분 난이도

이 책에서 가장 중요한 말을 먼저 할게.

너는 코더가 아니야. 디렉터야.

코드를 짜는 건 AI가 해. 너는 "뭘 만들지" 방향을 정하고,
"이게 맞는지" 눈으로 확인하는 사람이야.
영화 촬영장에서 조감독이 촬영 계획을 짜면 디렉터가 "OK" 또는 "다시!"를 말하잖아.
그거랑 똑같아.

"뭘 원해"로 말해. "어떻게 해"로 말하지 마.

음식점에서 주방장한테 이렇게 말하면 이상하잖아:
"감자를 1mm로 채 썰어서 180도 기름에 3분 40초 튀겨줘."

그냥 "바삭한 감자튀김 주세요"라고 하면 돼.
주방장은 전문가니까, 최선의 방법을 알아서 쓸 거야.

AI한테도 똑같아. "How"(어떻게)를 말하면
AI가 더 좋은 방법을 생각할 여지를 없애버려.

실제로 우리가 41일 동안 겪은 것들을 정리하면 이렇게 돼:

How (이렇게 하면 안 돼)What (이렇게 해)
"partial(파셜, 부분 파일)로 쪼개야 하나?""이 파일 너무 커진 것 같은데 괜찮아?"
"해시에 이 필드 포함해야 해?""설정 바꿨는데 왜 반영이 안 돼?"
"Dictionary 대신 List 써야 하나?""이거 점점 느려지는 것 같은데?"
"isRandom 체크를 프로퍼티로?""Random이랑 Sequential이 다르게 동작하는데?"
"OnEnable에서 초기화해야 하나?""씬 열 때마다 이게 초기화 안 돼 있어"
"캐시를 무효화해야 하나?""아까 바꾼 게 반영 안 된 것 같아"
"static(스태틱, 고정값) 필드를 리셋해야 하나?""유니티 다시 시작하면 이전 값이 남아 있어"
"Undo(언두, 되돌리기) 순서를 바꿔야 하나?""Ctrl+Z 하면 이상하게 돌아가"
"이벤트 구독을 해제해야 하나?""이 기능 끄고 다른 거 했는데 아직 반응해"
"A*(에이스타, 최단 경로 알고리즘) 반복 상한을 줄여야 하나?""가끔 화면이 뚝 끊겨"

왼쪽은 주방장한테 레시피를 지시하는 거야. 오른쪽은 손님이 느낀 걸 말하는 거야.
오른쪽으로 말하면 AI가 모든 기술적 가능성을 열어두고 최선의 답을 찾아와.

How를 먼저 말하면 AI의 해결 범위를 좁히는 독약이야.
"partial(파셜)로 쪼개야 하나?"라고 물으면 AI는 "네/아니오"만 대답해.
"파일이 너무 커진 것 같아"라고 하면 AI가 "partial(파셜) 분할이 좋겠어, 이유는..."까지 설명해줘.

찜찜함 = 최고의 버그 리포트

건물에서 연기 냄새가 나. 불은 안 보여.
이때 "괜찮겠지" 하고 넘기면? 큰일 날 수 있어.
"뭔가 타는 냄새 나는데?" 한 마디만 하면? 소방관이 알아서 찾아줘.

찜찜함은 연기 탐지기야. 불이 어디 있는지 몰라도 돼.
냄새만 맡으면 돼.

실제로 Bobby가 "찜찜하다"고 한 마디 한 것들이 어떻게 됐는지 봐:

Bobby의 한 마디결과
"WP(더블유피, 웨이포인트 — 경유점)가 안 타"Segment Stitching(세그먼트 스티칭, 경로 조각 이어붙이기) 아키텍처 탄생 (15만 줄 시스템의 핵심 구조 변경)
"에이전트가 GW에서 멈춰"A*(에이스타) 예산 시스템 + 좀비 가드 추가
"도로가 안 보여"corridor(코리도, 복도 경로) excludedCells 버그 발견
"이거 점점 느려지는 것 같아"A*(에이스타) 75회 호출을 1회로 줄이는 최적화
"코드가 커져서 걱정돼"파일 구조 전면 재설계 (SLV 파티션)

코드 한 줄 안 읽고도 16만 줄 시스템의 구조 결함을 찾아내는 디렉터.
이게 바로 너의 자리야.

디렉터와 조감독의 관계

디렉터가 "이 장면은 슬프게 가자"고 하면,
조감독이 "조명 톤 낮추고, 카메라 느리게, 음악 빼겠습니다"라고 해.
디렉터가 "카메라를 35mm로 바꾸고 3초 틸트업 해"라고 하면?
촬영감독보다 기술을 모르는 디렉터가 촬영을 제한하는 거야.

"What"을 말하면 크루가 최선의 "How"를 가져와.
"How"를 말하면 크루가 거기에 갇혀.

Q1. 이 중 What으로 말하는 건 뭘까?

A. "이 Dictionary를 ConcurrentDictionary로 바꿔줘"
B. "여러 에이전트가 동시에 지나가면 가끔 튕겨"
C. "for 루프를 foreach로 바꿔줘"

정답

B. "여러 에이전트가 동시에 지나가면 가끔 튕겨"가 What이야.
증상(느낌)을 말했지, 해결 방법(How)을 지정하지 않았으니까.
A와 C는 구체적인 코드 수정 방법을 지시하는 거야 — How.

Q2. Bobby가 "아까 바꾼 게 반영 안 된 것 같아"라고 했어. 이건?

A. 기술적 지시 (How)
B. 느낌 전달 (What)

정답

B. "반영 안 된 것 같아"는 느낌이야.
AI가 이걸 듣고 "캐시 문제인지, 해시 문제인지, 저장 순서 문제인지" 전부 확인할 수 있어.

Q3. "이벤트 구독을 해제해야 해?"라고 물었어. 뭐가 문제야?

정답

AI한테 해결책(How)을 먼저 제시한 거야.
"이 기능 끄고 다른 거 했는데 아직 반응해"라고 하면
AI가 이벤트 구독 해제가 답인지, 아니면 다른 문제인지까지 확인해줘.
How를 말하면 AI는 "네, 해제하세요"에서 끝나.

이 챕터 정리

  • 너는 코더가 아니라 디렉터야
  • "어떻게"(How)가 아니라 "뭘 원해"(What)로 말해
  • 찜찜함은 최고의 버그 리포트 — 연기만 맡으면 소방관이 불을 찾아줘
  • 2권은 "프롬프트 기법", 3권은 "마인드셋" — 기법보다 자세가 먼저야

AI 크루 싸움 붙이기 — 왜 AI 하나보다 둘이 나은지

Ch.2 AI 크루 싸움 붙이기

약 12분 난이도

AI 하나로도 충분하다고? 맞아, 충분할 때도 있어.
근데 프로젝트가 커지면 혼자선 놓치는 게 생겨.
그래서 우리는 AI를 두 명 붙였어.

프로덕션 크루: 디렉터 + 스카우터 + 촬영감독

영화 촬영을 생각해봐.

  • 디렉터 — "이런 분위기의 장면이 필요해" (방향 설정)
  • 스카우터(로케이션 헌터) — "로케이션 50곳 봤는데 여기 5곳이 괜찮아!" (넓게 탐색, 후보 제시)
  • 촬영감독(DP) — "5곳 중 조명/각도 확인해보니 진짜 쓸 만한 건 2곳이야" (정밀 검증)

스카우터가 없으면 촬영감독이 어디서 찍을지 몰라.
촬영감독이 없으면 스카우터가 찾은 장소가 진짜 괜찮은지 확인 못 해.
디렉터가 없으면 둘 다 방향을 잃어.

우리 팀에서는 이렇게 역할을 나눴어:

역할누가하는 일강점한계
디렉터 Bobby (너) 방향 설정 + 모니터로 확인 + "OK!/다시!" 감각, 직감, 눈 코드 세부
스카우터 Antigravity 520파일 전체 훑기 + 후보 리스트 작성 넓은 시야 오탐 80%
촬영감독 Claude 후보 검증 + 진짜만 남기기 + 코드 수정 정밀 촬영 시야 좁음

스카우터가 추천한 10곳 중 8곳은 실제로 가보면 안 맞아.
이게 정상이야. 넓게 보는 게 스카우터 역할이니까.
촬영감독이 "진짜 쓸 만한 건 2곳이야"라고 걸러주는 게 핵심이야.

Antigravity의 역할

Claude가 "실행팀"이라면, Antigravity는 "감시팀"이야.
Claude가 코드를 쓰면 Antigravity가 검증하고,
Antigravity가 설계를 제안하면 Claude가 구현해.
두 AI가 서로를 견제하니까 품질이 올라가는 거야.
"같은 회사 감사팀과 실행팀"이라고 생각하면 돼.

실제 숫자로 보자

우리가 41일 동안 실제로 겪은 숫자야:

상황처음 보고진짜 문제헛소리 비율
규칙 위반 스캔 (SSoT(에스에스오티, 단일 원본))5건1건80%
성능 병목 스캔18곳3곳83%
전체 코드 공방136건20건85%
2차 전수 스캔 (393파일)112건18건84%

AI 혼자 "이거 문제야!"라고 한 것 중 진짜 문제는 10개 중 2개뿐이야.
나머지 8개를 걸러내는 게 교차검증이야.

병원에서 진단받는 것과 같아.

의사 한 명이 X-Ray 보고 "이상 소견 5개"라고 해.
다른 전문의가 확인하면 "진짜 문제는 1개, 나머지는 정상 변이"라고 해.
세컨드 오피니언 없이 바로 수술했으면? 4번 헛수술한 거야.

AI 하나밖에 없을 때 — 1인 크루

스카우터 없이 촬영감독(Claude)이 혼자 로케 헌팅 + 촬영을 겸하는 거야.
그래도 교차검증 할 수 있어. 새 세션을 열면 돼.

같은 Claude라도 세션이 다르면 다른 각도로 봐.
첫 번째 세션에서 "이거 문제야"라고 한 걸
두 번째 세션에서 "진짜야? 확인해봐"라고 시키면 돼.

우리도 실제로 이렇게 했어:

  • 5개 세션을 동시에 열어서 각각 다른 영역을 스캔
  • 한 세션이 찾은 걸 다른 세션이 검증
  • 결과: 한 세션이 놓친 CRITICAL 버그를 다른 세션이 발견

5건이 1건이 되는 이야기

실제로 있었던 일이야.

Antigravity가 코드 전체를 훑고 와서 이렇게 보고했어:
"규칙 위반 5건 발견했습니다."

Bobby가 이걸 Claude에게 넘겼어. "진짜야? 확인해봐."

Claude가 5건을 하나하나 뜯어봤어:

  • 1번: "이건 이미 다른 곳에서 보호하고 있어. 정상."
  • 2번: "에디터에서만 돌아가는 코드야. 런타임 문제 아님."
  • 3번: "주석에 '의도적'이라고 써 있어. 설계 의도야."
  • 4번: "Unity 엔진이 알아서 처리하는 부분이야."
  • 5번: "이건 진짜 문제야. 수정 필요."

Antigravity가 5건을 찾은 건 잘한 거야. 그 중에 진짜 1건이 있었으니까.
근데 5건 전부 수정했으면? 4건은 멀쩡한 코드를 건드린 거야.
멀쩡한 코드를 건드리면 새 버그가 생겨.

화재 경보가 울렸을 때.

경보가 5번 울렸어. 매번 소방차가 와.
확인해보니 4번은 요리 연기, 1번만 진짜 불이었어.
그렇다고 경보기를 끄면 안 돼. 진짜 불을 놓치니까.
경보가 울리면 확인하되, 바로 물 뿌리지 마.

Bobby가 할 건 딱 하나야

AI 둘이 논의한 다음에 Bobby한테 올라와.
Bobby는 한 마디만 하면 돼:

"진짜야?"

이 한 마디가 80%의 헛소리를 걸러내는 마법의 말이야.
AI가 "확실합니다!"라고 해도 한 번 더 물어봐. "진짜야?"
그러면 AI가 다시 한번 확인하고, 확신이 없으면 솔직하게 말해.

그리고 최종 확인은 유니티에서 눈으로. 이건 기계가 절대 못해.
"느낌이 맞나?"를 판단하는 건 디렉터의 특권이야.

AI가 코드에서 버그 10개를 찾았대. 진짜는 몇 개일까?

정답

2~3개. 우리 경험상 AI 단독 분석의 정확도는 15~20%야.
10개 중 7~8개는 멀쩡한 코드를 "수상하다"고 신고한 거야.
그래서 반드시 교차검증(다른 AI나 새 세션)이 필요해.

판단을 기록하는 구체적 방법(확정/조건부/오탐/미확정)은 2권 Ch.3에서 배울 수 있어.

이 챕터 정리

  • 프로덕션 크루: 디렉터(방향) + 스카우터(넓게 탐색) + 촬영감독(정밀 검증)
  • AI 단독 분석은 10개 중 8개가 헛소리 — 교차검증 필수
  • AI 하나만 있어도 새 세션으로 교차검증 가능
  • Bobby가 할 건 "진짜야?" 한 마디 + 유니티에서 눈으로 확인

AI 크루가 일할 수 있는 환경 만들기

Ch.3 우리 팀 규칙서 만들기

약 8분 난이도

새 알바가 첫날 출근하면 뭘 받아? 매뉴얼.
"음료는 이렇게 만들어", "재고는 여기 확인해", "손님 불만은 이렇게 처리해."

AI한테도 매뉴얼이 필요해. 그게 CLAUDE.md야.
이 파일 하나가 AI의 행동을 결정해.

규칙 없던 시절 vs 있은 후

교통 규칙이 없는 도시를 상상해봐.
신호등도 없고, 우선순위도 없어. 차가 마구 움직여.
사고가 매일 터져.

규칙을 만들면? "빨간불에 멈춰", "우회전은 신호 확인 후."
사고가 확 줄어. 근데 규칙이 너무 많으면? "좌회전은 화요일만 허용"
아무도 안 지켜.

우리 프로젝트에서 실제로 일어난 일:

상황규칙 없던 시절규칙 있은 후
안전장치 빠뜨리기 (delayCall(딜레이콜, 지연 실행 명령)) 4건 동시 발견 (감사 때) 0건 (규칙이 자동 방지)
스타일 일괄 변경 963건 마이그레이션 → UI 전체 깨짐 → 86파일 되돌림 규칙 53으로 금지 → 재발 0
모드 전환 후 상태 잔류 6건 연속 발생 (#3~#7) EnterMode(엔터모드, 모드 진입)/ExitMode(엔터모드/엑싯모드, 모드 진입/퇴장) 쌍 규칙 → 거의 0
같은 문제 반복 접근 도로 오버레이 5세션, 입력 문제 8번 memory에 "이 길은 막힘" 기록 → 반복 감소

좋은 규칙 vs 나쁜 규칙

좋은 규칙나쁜 규칙
"파일이 500줄 넘으면 쪼개""파일은 무조건 200줄 이하"
"모드 끝낼 때 상태 변수 정리해""전역 변수 금지"
"D3D12(디쓰리디트웰브, 최신 그래픽 처리 방식) 쓰지 마 (화면 깨짐)""새 기술 시도하지 마"
"인라인 스타일 일괄 변경 금지""인라인 스타일 절대 금지"

좋은 규칙의 특징:

  • 구체적 — "500줄"이라는 숫자가 있어
  • 이유가 있어 — 왜 안 되는지 설명돼 있어
  • 지킬 수 있어 — 너무 빡빡하지 않아

나쁜 규칙의 특징:

  • 너무 추상적 — "좋은 코드를 짜라" (뭐가 좋은 건데?)
  • 이유가 없어 — "그냥 이렇게 해" (왜?)
  • 현실 무시 — A*(에이스타) 경로 계산은 변수만으로 500줄 넘어

큰 그물 + 촘촘한 그물

우리 규칙은 2계층 구조야.

큰 그물: 절대 규칙 5개 (어떤 상황에서도)

  1. 기능은 최소 코드로 동작 먼저 — 안 시킨 거 추가하지 마
  2. 수정 → 테스트 → 확인 → 다음 — 한꺼번에 여러 개 하지 마
  3. 모드 전환은 Enter + Exit 쌍으로 — 안 쓰는 상태 남기지 마
  4. GC(가비지) 만들지 마, 캐시 써 — 성능은 기본
  5. 보호 파일은 규칙 번호 명시 후 수정 — 함부로 건드리지 마

촘촘한 그물: 상황별 규칙 (구체적 지뢰)

  • "D3D12(디쓰리디트웰브) 쓰지 마" (규칙 23) — 화면 깨짐
  • "delayCall(딜레이콜) 콜백 첫 줄에 null(널, 값 없음) 체크" (규칙 E4) — 크래시 방지
  • "해시에 새 필드 추가했으면 캐시도 갱신" — 설정 반영 안 되는 버그 방지
  • "인라인 스타일 일괄 마이그레이션 금지" (규칙 53) — UI 파괴 방지

왜 2계층이냐면, 큰 원칙만 5개 있으면 구체적 지뢰를 못 막아.
"좋은 코드를 짜라"로는 D3D12(디쓰리디트웰브) 금지를 못 잡아.
반대로 구체적 규칙만 53개 있으면 새로운 상황에서 판단 기준이 없어.

규칙 하나 만들어보기

만약 이런 문제가 반복된다면:

"Ctrl+Z 하면 이상하게 돌아가." (3번 발생)

규칙 만들기 템플릿

문제: [Ctrl+Z 하면 UI가 이상하게 복원됨]

원인: [값을 바꾸기 전에 Undo(언두) 기록을 안 남김]

규칙: [값 바꾸기 전에 반드시 Undo(언두).RecordObject 호출]

확인법: [수정 후 Ctrl+Z 3번 해보기]

이걸 CLAUDE.md에 적어두면, AI가 다음부터 자동으로 Undo(언두) 기록을 남겨.

같은 문제가 3번 터졌어. 어떻게 해?

A. 매번 수정한다
B. 규칙을 만들어서 CLAUDE.md에 적는다
C. AI를 바꾼다

정답

B. 같은 문제가 3번 터지면 패턴이야.
규칙으로 만들면 AI가 앞으로 자동으로 피해.
우리 프로젝트에서도 delayCall(딜레이콜) 가드를 규칙화한 뒤로 재발 0건이야.

이 챕터 정리

  • CLAUDE.md = 새 직원 첫날 매뉴얼
  • 좋은 규칙: 구체적 + 이유 있음 + 지킬 수 있음
  • 2계층 구조: 절대 원칙 5개 + 상황별 구체 규칙
  • 같은 문제 3번 → 규칙 만들기

AI가 안 까먹게 하기 — 세션 관리와 인수인계 노트

Ch.4 AI가 안 까먹게 하기

약 8분 난이도

2권에서 "Claude는 천재인데 기억상실"이라고 했어.
3권에서는 그래서 어떻게 관리하는지를 알려줄게.

"파일 긴 것 같아" 한 마디면 끝

서랍이 하나인 책상을 상상해봐. 서류가 500장 들어 있어.
"계약서 어딨지?" 찾으려면 500장을 다 뒤져야 해.

서랍을 5개로 나누면? "계약서는 세 번째 서랍"이라고 알면 100장만 보면 돼.
내용은 똑같아. 정리만 달라진 거야.

코드 파일도 똑같아. 한 파일이 800줄이 넘으면 AI도 헷갈려.
이때 너는 이렇게만 말하면 돼:

"이 파일 너무 커진 것 같은데 괜찮아?"

그러면 AI가 알아서 쪼개줘. 변수는 한 곳에 모으고, 기능별로 나눠서.
우리 프로젝트에서도 800줄짜리 파일 5개를 이렇게 쪼갰어.
Bobby는 코드를 안 봤어. "커 보여"라고만 했어.

인수인계 노트 = memory/

편의점에서 교대 근무한다고 생각해봐.
밤 근무자가 퇴근하면서 아침 근무자한테 메모를 남겨:
"3번 냉장고 고장. 수리 업체 내일 옴. 우유는 2번으로 옮겨놨음."

이 메모가 없으면? 아침 근무자가 처음부터 다시 파악해야 해.
"어? 3번 냉장고 왜 비어 있지?" → 고객한테 물어보고 → 업체 전화하고...
이미 해결된 문제를 또 해결하는 거야.

memory/ 폴더가 바로 이 인수인계 노트야.
AI 세션이 바뀔 때마다 이전에 뭘 했고, 뭘 배웠는지 적혀 있어.

우리 프로젝트에서 memory/에 뭘 적어뒀냐면:

  • 교훈 — "이 방법은 안 되더라. 6일 날리고 결국 제거했어"
  • 규칙 — "delayCall(딜레이콜)에 null(널) 체크 넣어야 돼"
  • 에피소드 — "WP(더블유피) 경로 경유 실패 → 이렇게 해결했어"
  • 피드백 — "Bobby가 이런 피드백 줬어"

이게 없었으면? 도로 오버레이 문제를 5세션에서 반복하는 게 10세션이 됐을 거야.

세션 끊기 — 디렉터의 판단

세션을 언제 끊는지(2번 반복 / 합의 위반 / 5파일 수정)는 2권 Ch.3에서 자세히 다뤘어.
여기서는 디렉터가 왜 끊는 결정을 직접 해야 하는지만 짚을게.

AI는 이전 시도에 갇히는 경향이 있어.
한 세션에서 "이 방법으로 해보자" → 실패 → "다시 해보자" → 또 같은 방법...
AI 스스로는 "내가 갇혔다"를 잘 모름. 끊는 건 디렉터만 할 수 있는 판단이야.

실제 사례: Theta*(세타스타, 시야 기반 길찾기) 알고리즘을 6일 동안 붙잡았어.
세션을 바꾸고 나서야 "데이터 구조를 바꾸면 되잖아?"라는 답이 나왔어.
결국 Theta*(세타스타)는 전부 삭제하고, 더 간단한 방법으로 해결했어.
끊는 타이밍을 놓치면 6일을 날려.

세션 끊어야 할까? 상황 판단해봐.

상황 1: AI가 "이건 못 고치겠어"라고 했다가 "다시 해볼게"라고 하는데, 같은 코드를 또 보고 있어.

정답

끊어야 해. 같은 문제를 같은 방법으로 2번째 시도하고 있어.
새 세션에서 다른 접근법을 찾을 확률이 높아.

상황 2: AI가 파일 3개를 수정했고, 아직 테스트 안 했어.

정답

아직 안 끊어도 돼. 먼저 유니티에서 테스트해.
문제없으면 계속 진행, 문제 있으면 그때 판단해.

상황 3: AI가 "이 파일은 건드리지 말자"라고 합의했는데, 방금 그 파일을 수정했어.

정답

끊어야 해. 합의를 위반했으면 이 세션의 맥락이 꼬인 거야.
수정을 되돌리고 새 세션에서 다시 시작해.

이 챕터 정리

  • "파일 긴 것 같아" 한 마디면 AI가 알아서 쪼개줘
  • memory/ = 교대 근무 인수인계 노트
  • 세션 끊기 신호는 2권 참조 — 핵심은 "끊는 판단은 디렉터만 할 수 있다"
  • AI는 이전 시도에 갇혀 — 끊는 타이밍을 놓치면 며칠을 날려

Ch.5 프로젝트 시작부터 끝까지

약 15분 난이도

지금까지 마인드셋이랑 세팅을 배웠어.
이제 실제로 프로젝트를 어떻게 진행하는지 처음부터 끝까지 보여줄게.

4단계 파이프라인

모든 기능은 이 4단계로 만들어져:

1

"이런 기능 원해"

BOBBY

UX(유엑스, 사용자 경험) 목표만 말해. 코드 용어 없이.
"웨이포인트 드래그하면 영역이 실시간으로 커졌으면 좋겠어. 버벅거리면 안 돼."

2

로케이션 헌팅 + 후보 리스트

스카우터 (또는 Claude 겸임)

코드 전체를 훑고 "이 파일들을 수정해야 해"라는 리스트를 만들어.
"수정 대상: A, B, C 파일. 부작용 주의: D 파일."

3

후보 검증 + 촬영(구현)

촬영감독 (Claude)

후보 파일을 하나하나 뜯어보고, 진짜만 남기고, 코드를 짜.
"5건 중 3건은 정상. 나머지 2건 수정 완료."

4

유니티에서 확인

BOBBY

Play(플레이, 실행) 버튼 누르고, 마우스로 만져보고, 눈으로 확인.
"됐다!" → 커밋. "0.1초 밀려" → 스카우터에게 성능 재스캔 요청.

집 리모델링이랑 같아.

1단계: "부엌을 밝게 만들고 싶어" (집주인 = Bobby)
2단계: "전기배선 확인했습니다, 이 벽은 못 뚫어요" (설계사 = 스카우터)
3단계: "조명 설치 완료, 콘센트 위치 확인해주세요" (시공팀 = 촬영감독)
4단계: "오, 밝아졌다! 근데 스위치가 너무 높아" (집주인이 눈으로 확인)

스카우터 없이 촬영감독이 겸업할 때

AI가 하나뿐이면? 2단계를 Claude가 겸임하면 돼.

  1. Bobby: "이런 기능 원해"
  2. Claude: 스스로 코드 훑고 + 수정 대상 찾고 + 구현
  3. Bobby: 유니티에서 확인

이때 중요한 건 "진짜야?" 한 번 더 물어보는 거야.
스카우터가 없으니까 Claude가 혼자 판단한 거거든.
"이게 최선이야? 다른 방법은 없어?"라고 한 번만 물어봐.

가상 시뮬레이션: 오토타일 기능 만들기

실제로 해보자. "타일이 주변 타일에 맞게 자동으로 바뀌는 기능"을 만든다고 치자.

1단계: Bobby가 말한다

"타일을 놓으면 주변 타일이랑 자동으로 이어졌으면 좋겠어. 풀밭 타일 놓으면 옆에 있는 풀밭이랑 자연스럽게 연결되는 거."

2단계: Claude가 분석한다

"타일맵 시스템을 확인했어. Rule Tile이라는 게 Unity에 있는데, 이걸 쓰면 자동으로 주변 타일에 맞게 바뀌어. 수정할 파일은 TilePainter.cs랑 TileCatalog.cs야."

3단계: Bobby가 확인한다

"오, 된다! 근데 대각선이 좀 어색한데?"

4단계: 반복

Claude가 대각선 규칙 조정 → Bobby 확인 → "됐다!" → 커밋

이게 전부야. 복잡하지 않아.

복잡한 시뮬레이션: "두 AI가 서로 모순되는 의견을 낼 때"

Claude: "이 함수는 삭제해도 안전합니다."
Antigravity: "이 함수는 동적 호출로 쓰이고 있어서 삭제하면 런타임 에러납니다."

디렉터 판단: Antigravity에게 증거를 요구해.
"동적 호출이 어디서 일어나는지 보여줘." 증거 있으면 Antigravity가 맞고, 없으면 오탐.

커밋: "세션은 수정만, HEAD(헤드, 조율 세션)가 커밋"

프로젝트가 커지면 여러 세션이 동시에 일할 때가 있어.
이때 커밋(세이브 포인트)을 아무나 찍으면 충돌이 나.

우리는 3번 실패한 끝에 이 규칙을 만들었어:
"각 세션은 코드 수정만 해. 세이브 포인트는 메인 세션이 찍어."

편의점으로 치면, 알바생 여러 명이 동시에 금전 등록기를 만지면 안 되잖아.
한 명만 등록기 담당, 나머지는 물건 정리만.

비용 감각: AI 크루는 무료가 아니야

멀티세션 = 비용 × 세션 수. 5개 세션을 동시에 돌리면 5배.

그래서 "정말 필요할 때만" 멀티세션으로. 일상적인 수정은 단일 세션이면 충분해.

B. CLAUDE.md를 만든다
C. 규칙 50개를 먼저 정한다

정답

B. CLAUDE.md를 먼저 만들어. 프로젝트 이름, 기술, 절대 규칙 3개만.
규칙은 처음에 3개면 충분해. 문제가 터질 때마다 하나씩 추가하면 돼.

이 챕터 정리

  • 4단계 파이프라인: 원해 → 스캔 → 구현 → 확인
  • AI 하나면 Claude가 겸임. "진짜야?" 한 번 더 물어보기
  • 규모에 따라 세팅을 키워가 — 처음부터 풀세팅 필요 없음
  • Day 1: CLAUDE.md + 첫 기능 + 확인 + 커밋
  • 커밋은 메인 세션이 전담 — 여러 명이 동시에 세이브하면 충돌

Ch.6 우리가 배운 것들

약 10분 난이도

41일 동안 430+커밋을 하면서 같은 종류의 실수가 반복됐어.
정리해보니 5가지 유형으로 묶이더라.

우리가 가장 많이 겪은 문제 5가지

1. "꺼놓고 나왔는데 켜져 있었어" — 상태 잔류

방을 나갈 때 불을 안 끈 거야.
다음에 다른 방에 들어갔는데, 이전 방 불이 아직 켜져 있어.
"어? 나 여기 불 안 켰는데 왜 밝지?"

기능 A를 켜고 쓰다가 기능 B로 넘어갔는데,
기능 A의 설정값이 아직 남아 있어서 기능 B가 이상하게 동작하는 거야.

우리 사례: 페인트 모드에서 나왔는데 페인트 상태가 남아서 6번 연속 발생 (#3~#7).

해결: "방을 나갈 때 반드시 불 끄기" 규칙 — EnterMode(엔터모드)/ExitMode(엔터모드/엑싯모드) 쌍.

2. "여기선 되는데 저기선 안 돼" — 이중 오케스트레이션

매장에 매니저가 두 명 있어. 둘 다 "내가 총괄이야"라고 생각해.
한 명이 "오늘 할인 행사"라고 했는데, 다른 한 명은 몰라.
결과? 어떤 계산대는 할인이 되고, 어떤 계산대는 안 돼.

같은 기능을 두 곳에서 따로 관리하면 서로 타이밍이 안 맞아.

우리 사례: 기즈모(보조선) 리빌드를 두 파일에서 따로 판단 → 5세션 소비.

해결: "매니저는 한 명만" — 결정은 한 곳에서만.

3. "설정 바꿨는데 반영 안 돼" — 캐시 누락

메모장에 전화번호를 적어뒀어. 상대방이 번호를 바꿨어.
근데 메모장은 안 고쳤어. 예전 번호로 전화하면 안 받겠지.

설정을 바꿨는데 "메모해둔 값"(캐시)이 업데이트 안 된 거야.

우리 사례: 경로에 영향 주는 필드 7개를 추가했는데 해시(캐시 키)에 안 넣음 → 설정 바꿔도 경로 안 바뀜.

해결: "새 설정 추가하면 메모장도 같이 고쳐" — 해시 체크리스트.

4. "갑자기 터져" — 가드 누락

엘리베이터에 문 닫힘 센서가 없어.
사람이 끼어도 그냥 닫혀. 사고 나.

예외 상황에 대한 안전장치가 없어서 갑자기 크래시 나는 거야.

우리 사례: 씬 전환 후 객체가 사라졌는데, 예약된 콜백이 "사라진 객체"를 쓰려고 함 → 크래시 4건 동시 발견.

해결: "콜백 첫 줄에 안전장치" — null(널) 체크 규칙화.

5. "마우스가 안 먹어" — 입력 차단

리모컨 배터리가 나갔는데, 리모컨 탓인지 TV 탓인지 모르는 거야.
아무 버튼이나 눌러봐도 반응이 없어.

유니티 에디터에서 마우스 클릭이나 패닝이 갑자기 안 되는 거야.

우리 사례: Alt+Tab으로 다른 프로그램 갔다 오면 마우스 패닝 불가 — 8번 수정.

해결: "입력 관리는 한 곳에서" — 중앙 집중화.

디렉터도 실수한다

"과잉 수정 충동"은 디렉터의 가장 큰 적이야.

솔직하게 말할게. 이 방법도 만능은 아니야.

Theta*(세타스타) 알고리즘 사건:
