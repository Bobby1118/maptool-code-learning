Coroutine — 잠깐 멈추고 다음에 이어서

Coroutine(코루틴)은 "여기서 잠깐 멈추고 다음 프레임에 이어서"를 할 수 있는 특별한 함수야. yield return이라는 키워드를 만나면 "여기까지만 하고, 다음에 계속할게"라는 뜻이야.

코드에서 IEnumerator나 yield return을 보면 코루틴인 거야. 지금은 "시간을 쪼개서 실행하는 함수"라고만 기억해두면 돼.

Q1: Awake와 Start(스타트)의 차이는?
정답 보기
Awake = 태어나자마자 (다른 애들이 아직 준비 안 됐을 수 있어). Start(스타트) = 모든 준비 끝나고 첫 프레임.
Q2: OnEnable에서 등록한 이벤트를 OnDisable에서 해제 안 하면?
정답 보기
오브젝트가 꺼져도 이벤트가 계속 남아있어. 없는 오브젝트한테 알림이 가니까 에러나.

LateUpdate / FixedUpdate

LateUpdate() = Update가 끝난 뒤 실행. 카메라 따라가기에 쓰여.

FixedUpdate() = 물리 엔진 타이밍으로 실행. 물리 연산할 때.

MapTool에서는 LateUpdate보다 Update가 훨씬 많아.

Ch.18 정리

Unity 시간표: Awake → OnEnable → Start(스타트) → Update. OnEnable/OnDisable은 항상 짝으로.

Unity에는 두 개의 세계가 있어.

Editor(에디터) = 편집할 때 (Play(플레이, 실행) 안 누른 상태). 타일 놓고, 설정 바꾸고, 인스펙터 만지는 그 세계.

비유하면 영화 "편집실"과 "극장"이야. 편집실에서 쓰는 도구(가위, 편집기)를 극장에 가져갈 수 없어.

Singleton — 전 세계에 딱 하나

전 세계에 딱 하나. 유니콘 같은 거야. 근데 코드에선 흔해.

Singleton은 "한 명만 있는 반장"이야.

Factory — 주문에 따라 만들어주는 공장

Factory는 "커스텀 주문을 받는 공장"이야.

"빨간 자동차 만들어줘" 하면 빨간 자동차를, "파란 트럭 만들어줘" 하면 파란 트럭을 만들어줘. 주문에 따라 다른 물건이 나오는 패턴.

MapTool에서는 프리팹(미리 만든 오브젝트 틀)을 만들 때 이 패턴을 써. "타일 프리팹 만들어줘"와 "NPC(엔피씨) 프리팹 만들어줘"가 같은 공장을 통해 나오는 거야.

코드를 외울 필요 없어. "Factory"라는 이름이 보이면 "아, 뭔가를 주문대로 만들어주는 곳이구나" 하면 돼.

Factory는 이름만 알면 돼. 다음은 캐시 — 이건 코드로 볼 거야.

캐시 — 한 번 찾으면 메모해두기

캐시는 메모장이야.

전화번호를 매번 전화번호부에서 찾는 대신, 한 번 찾으면 종이에 적어두잖아. 다음에는 종이만 보면 돼. 그게 캐시야.

방금 봤던 Singleton 코드에서 캐시를 이미 봤어:

도미노 비유:

1 AgentSystem과 Occupancy(오큐펀시)가 양방향으로 연결돼 있어. 경로 탐색(Navigation)은 AgentSystem 안에 있으니까, 별도 시스템이 아니라 같은 부서 내부 팀이야.

Ch.20D — 캐시의 생과 사: 데이터가 오래되면 어떻게 되나

약 12분 난이도 새 개념: 캐시 파이프라인 9개

Ch.20B에서 데이터의 흐름을 봤어.
이번에는 데이터가 흐르다가 상하거나, 새로 만들어지거나, 화면에 뿌려지는 과정을 볼 거야.

식당을 떠올려봐. 주방에서 음식을 만들고, 홀에서 손님한테 나가잖아?
코드도 똑같아. 데이터를 만들고 → 저장하고 → 다시 쓰고 → 화면에 보여주는 과정이 있어.

식당 비유 — 9가지 개념 한 장 정리

1. State (시스템 현재 상태) = 주방의 현황판. "지금 영업 중인지, 점심 메뉴인지 저녁 메뉴인지."
2. Cache(캐시, 임시 저장) (계산 결과 임시 저장) = 미리 만들어둔 반찬. "주문 올 때마다 새로 만들면 느리니까."
3. Invalidate (이전 값 못 믿음 표시) = "반찬 상했어!" 표지판. 먹으면 안 되니까 표시만 해두는 거.
4. Rebuild (실제로 다시 계산) = 반찬 새로 만들기. 표지판 보고 진짜 다시 요리하는 거.
5. Sync (한쪽을 다른 쪽에 밀어넣기) = 주방 메뉴판과 홀 메뉴판 맞추기. 한쪽이 바뀌면 다른 쪽도 갱신.
6. Repaint (화면만 다시 그리기) = 메뉴판 먼지 닦기. 내용은 안 바뀌어, 보이는 것만 새로 칠하는 거.
7. Consumer(컨슈머, 값을 쓰는 쪽) (값을 써먹는 코드) = 실제로 주문한 손님. 반찬을 직접 먹는 사람.
8. Stale (오래된 값) = 유통기한 지난 식재료. 아직 버려지진 않았지만 쓰면 안 돼.
9. Gate (기능을 통째로 켜고 끄기) = 영업 중/준비 중 표지판. 걸려 있으면 손님이 들어와도 주문 안 받아.

데이터 파이프라인 — 도미노 순서

이 9가지는 따로 노는 게 아니야. 순서가 정해져 있어:

11 이벤트 발생 // "타일이 바뀌었어!" 알림이 와 22 → Invalidate // "반찬 상했어!" 표시 33 → Rebuild // 반찬 새로 만들기 44 → Sync // 주방 메뉴판 → 홀 메뉴판 맞추기 55 → Repaint // 화면 새로 칠하기 66 → Consumer(컨슈머) // 손님이 먹음!
1 이벤트 = 뭔가 바뀌었다는 알림 (Ch.10에서 배웠지?)
2 Invalidate = "이전 값은 못 믿어"라고 표시만 해. 아직 새로 만들진 않아.
3 Rebuild = 진짜로 다시 계산해. CPU가 일하는 단계.
4 Sync = 계산 결과를 필요한 곳에 복사해. (Editor(에디터) → Runtime(런타임), 또는 반대)
5 Repaint = 화면을 새로 그려. 데이터는 안 바뀌어!
6 Consumer(컨슈머) = 최종적으로 그 값을 쓰는 코드.

이 중 하나라도 빠지면 버그가 돼.

  • Invalidate만 하고 Rebuild 안 하면? → "상했다고 표시만 하고 새 반찬을 안 만든" 상태. Stale(유통기한 지남)이야.
  • Rebuild는 했는데 Sync 안 하면? → 주방에선 새 메뉴인데 홀에선 옛날 메뉴판. 손님이 딴 걸 시켜.
  • Repaint만 하면? → 메뉴판을 닦았지만 내용은 그대로야. 겉만 새거, 속은 옛날 거.

Bobby가 경험한 것: 리빌드 결정이 한 곳에서

예전에 기즈모 코드에서 Rebuild가 두 군데에서 일어났어.
A에서 Rebuild 했는데 B에서 또 Rebuild → 결과가 뒤집힘 → 5세션 디버깅.
파이프라인 순서를 지키지 않으면 이런 일이 생겨.

보너스: 해시 — 사진 대신 지문 비교

diff(디프) 읽기 — 빨간 줄과 초록 줄

Ch.7 사이드바에서 diff(디프) 형식을 잠깐 봤었지?
-는 삭제, +는 추가.
이제 "형식"이 아니라 "의미"를 읽는 거야.

diff(디프)는 "수정 전/후 비교"야.
빨간 줄(-)은 "이걸 지웠어".
초록 줄(+)은 "이걸 넣었어".
빨간 줄이랑 초록 줄을 나란히 보면 "뭐가 바뀌었는지" 보여.

GitHub에서는 빨간/초록 배경으로 보이고, 터미널에서는 -/+ 기호로 구분돼.

diff(디프) 읽기 3단계

#뭘 하나비유
1@@ 줄에서 위치 확인편지에서 "2페이지 3번째 줄 근처" 같은 것
2빨간 줄(-) 읽기"이게 없어졌어"
3초록 줄(+) 읽기"이게 새로 생겼어"

@@ -16,3 +16,5 @@에서 숫자는 뭐냐면: -16,3은 "원래 16줄부터 3줄이었는데", +16,5는 "수정 후에는 16줄부터 5줄이 됐어"라는 뜻이야. 숫자를 외울 필요 없어 — 위치만 확인하면 돼.

빨간 줄과 초록 줄을 비교하면 "뭐가 바뀌었는지"가 보여.

| 약 11p | 새 개념: 호출 체인, Ctrl+Click 따라가기

함수는 혼자 일하지 않아

Ch.8에서 함수(버튼)를 배웠잖아.
그때는 하나의 함수만 봤어.

근데 실제 코드에서는 함수가 다른 함수를 부르고, 그 함수가 또 다른 함수를 불러.
이 연쇄를 호출 체인이라고 해.

택배 추적이랑 똑같아.
출발지(주문) → 경유지(물류센터) → 도착지(집 앞).
코드도 마찬가지야. 시작점에서 경유지를 거쳐 최종 목적지까지 가.

호출 체인 예시 1: MapObjectInteraction

MapTool에서 캐릭터가 오브젝트와 상호작용하는 코드야.

1using System; 2using UnityEngine; 3 4namespace(네임스페이스) MapTool 5{ 6 public class MapObjectInteraction : MonoBehaviour(모노비헤이비어) 7 { 8 public event Action<MapObjectInteraction, GameObject(게임오브젝트)> OnInteract; 1 9 10 public void(보이드) Interact(GameObject(게임오브젝트) actor) 2 11 { 12 OnInteract?.Invoke(this, actor); 3 13 } 14 } 15}
1 OnInteract — 알림판(event). "상호작용이 일어나면 알려줘"라고 등록하는 곳
2 Interact — 이 함수가 불리면 상호작용이 시작돼
3 OnInteract?.Invoke(this, actor) — "알림판에 등록된 모든 함수를 실행해"

호출 체인을 그려보자

어딘가에서 Interact(actor)를 호출 →
  → OnInteract?.Invoke(this, actor) 실행 →
    → 등록된 콜백(알림판에 적어둔 함수들) 실행

여기서 핵심은 — Interact를 부르는 쪽과 OnInteract에 등록한 쪽이 다른 파일에 있을 수 있다는 거야.
호출 체인을 따라가려면 파일을 넘나들어야 해.

예측해봐: OnInteract에 아무도 등록하지 않았으면, Interact를 호출하면 어떻게 돼?

정답 보기

아무 일도 안 일어나. ?.가 "등록된 게 있으면 실행, 없으면 패스"니까. 에러도 안 나.

여기서 한 번 끊어도 돼. 다음은 프로퍼티 체인이야.

호출 체인 예시 2: SortingLayer 프로퍼티 체인

이번에는 함수가 아니라 프로퍼티(규칙 있는 상자)가 연쇄되는 경우야.

// MapToolConstants.cs에서 68public static(스태틱) string(스트링) SortingLayerGround => 1 69 MapToolSortingConfig.Instance != null(널) 2 70 ? MapToolSortingConfig.Instance.ground 3 71 : MapToolSortingConfig.DefaultGround; 4
1 SortingLayerGround를 읽으면...
2 먼저 MapToolSortingConfig.Instance가 있는지 확인 (null(널) 체크!)
3 있으면 → 설정 파일에서 값을 가져와
4 없으면 → 기본값 "Map_Ground"를 써

여기서 MapToolSortingConfig.Instance는 또 다른 파일에 있어. 따라가보자.

// MapToolSortingConfig.cs에서 42public static(스태틱) MapToolSortingConfig Instance 1 43{ 44 get 45 {
1 Instance — 프로퍼티(규칙 있는 상자). 읽을 때 자동으로 아래 코드가 실행돼.

이 안을 자세히 보자:

46 if (_instance != null(널)) return _instance; 1 47 if (_instanceSearched) return null(널); 2 48 _instanceSearched = true; 49 50 var configs = Resources.FindObjectsOfTypeAll<...>(); 3 51 if (configs.Length > 0) 52 _instance = configs[0]; 53 return _instance; 54 } 55}
1 이미 찾아놓은 게 있으면 바로 돌려줘 (캐시 = 메모)
2 이미 찾아봤는데 없었으면 다시 안 찾아 (헛수고 방지)
3 프로젝트 전체에서 SortingConfig 에셋을 검색

전체 체인을 이어보면

코드 어딘가에서 MapToolConstants.SortingLayerGround를 읽으면
  → MapToolSortingConfig.Instance를 확인하러 가고
    → _instance(캐시)가 있으면 바로 반환
    → 없으면 프로젝트에서 검색해서 찾고
  → 찾은 에셋의 .ground 값을 돌려줘
  → 에셋이 아예 없으면 기본값 "Map_Ground"

이렇게 점 하나(.) 뒤에 또 다른 세계가 있는 거야. 호출 체인을 따라가면 코드가 실제로 어떻게 동작하는지 보여.

실전 팁: 에디터에서 따라가기

이 세 가지면 호출 체인을 따라갈 수 있어. VSCode 기준이야. Rider나 다른 에디터도 비슷한 기능이 있어.

부록 C — 자립도 체크리스트

서문에서 Level 0~5를 소개했지? 여기 각 레벨의 체크리스트가 있어.
하나씩 체크해가면서 내가 어디쯤 왔는지 확인해봐.

Level 0 → Level 1: "지도를 읽을 수 있다"

이걸 할 수 있으면 Level 1이야.

  • ☐ 에러 메시지에서 파일명 + 줄번호를 뽑아낼 수 있다 (→ Ch.23)
  • ☐ Scripts 폴더와 Editor(에디터) 폴더의 차이를 안다 (→ Ch.19, Ch.20A)
  • ☐ namespace MapTool vs namespace MapToolEditor 구분할 수 있다 (→ Ch.19, Ch.20C)
  • ☐ [SerializeField]가 뭔지 알고, Inspector(인스펙터) 값과 코드 변수를 연결할 수 있다 (→ Ch.6)

Level 1 → Level 2: "변수 이름으로 추측"

이걸 할 수 있으면 Level 2야.

  • ☐ 변수명 읽고 역할을 추측할 수 있다 (_lastBrushCell = 마지막 브러시 셀) (→ Ch.3)
  • ☐ if, return, null 같은 기본 키워드의 의미를 안다 (→ Ch.7)
  • ☐ 코드에서 // 주석을 읽고 맥락을 파악할 수 있다 (→ Ch.1)
  • ☐ partial(파셜) class가 "설계도를 여러 장으로 나눈 것"임을 안다 (→ Ch.20C)

Level 2 → Level 3: "설정값 직접 찾기"

이걸 할 수 있으면 Level 3이야.

  • ☐ 상수값/설정값을 코드에서 직접 찾을 수 있다 (→ Ch.3, Ch.4)
  • ☐ 간단한 if 조건을 읽을 수 있다 (→ Ch.7)
  • ☐ Ctrl+F로 특정 변수가 어디서 쓰이는지 검색할 수 있다 (→ Ch.22)
  • ☐ 에러 메시지 보고 "이 파일의 이 변수가 null(널)이구나" 파악할 수 있다 (→ Ch.23)

Level 3 → Level 4: "코드 흐름 따라가기"

이걸 할 수 있으면 Level 4야.

  • ☐ 함수 호출 체인을 따라갈 수 있다 (A → B → C) (→ Ch.22)
  • ☐ 간단한 버그를 혼자 발견할 수 있다 (→ Ch.23 시뮬레이션)
  • ☐ Claude에게 "이 함수가 문제인 것 같은데, 맞아?" 수준의 질문을 할 수 있다
  • ☐ git diff(디프)를 읽고 변경 내용의 의미를 이해할 수 있다 (→ Ch.21)

Level 4 → Level 5: "에러 보고 문제 파일 특정"

이걸 할 수 있으면 Level 5야. 졸업! (교재 이후 실전에서)

  • ☐ 에러 스택 트레이스 전체를 읽고 원인 파일을 특정할 수 있다
  • ☐ 간단한 수정 (상수 변경, 조건 추가, null(널) 체크)을 혼자 시도할 수 있다
  • ☐ Claude에게 "사이드 이펙트 있을까?" 수준의 검증 요청을 할 수 있다
  • ☐ 새 기능 요청 시 구체적 파일/위치를 지시할 수 있다

이 교재를 다 읽으면 Level 3~4쯤이야.
Level 5는 실전에서 자연스럽게 올라가.
체크리스트에 체크가 하나씩 늘어갈 때마다, 그게 성장이야.
